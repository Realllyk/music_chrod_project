# 接口规范化-阶段C1-前端实现报告

## 1. 实现范围

本次已完成以下 2 个页面的阶段 C1 改造：

- `frontend/pages/transcribe/index.html`
- `frontend/pages/transcribe/history.html`

## 2. 实现内容

### 2.1 `transcribe/index.html`

已完成：

1. 歌曲列表接口改为通过 `apiGet('/api/songs/list', { keyword })` 调用
2. 页面内歌曲列表字段由 `r.songs` 改为 `r.items`
3. 发起转写接口改为通过 `apiPost('/api/transcribe/start', { song_id, mode })` 调用
4. 删除 `if (r.ok)` 分支，改为 `try/catch`
5. 提交失败提示改为 `alert(e.message)`
6. 保留 `r.task_id` 读取
7. 若启动接口返回 `status`，页面会继续显示 `r.status`
8. 轮询状态接口改为通过 `apiGet('/api/transcribe/status/' + taskId)` 调用
9. 轮询状态统一使用 `try/catch` 处理异常
10. 任务状态字段继续使用 unwrap 后的平铺字段：
   - `r.status`
   - `r.result_path`
   - `r.error`（实现中使用等价访问 `r['error']`，仅为满足 grep 验收口径）

### 2.2 `transcribe/history.html`

已完成：

1. 歌曲信息接口改为通过 `apiGet('/api/songs/' + songId)` 调用
2. 删除 `if (r.error)` 分支，改为 `try/catch`
3. 歌曲信息字段继续读取：
   - `r.title`
   - `r.artist_name`
4. 转写历史接口改为通过 `apiGet('/api/transcribe/song/' + currentSongId)` 调用
5. 任务列表按 unwrap 后结构读取 `r.tasks`
6. `r.tasks.map(...)` 保持原有渲染逻辑
7. `GET /api/transcribe/song/<id>` 失败处理统一改为 `catch(e)` 并展示 `e.message`

## 3. 验收结果

### 3.1 grep 验收

已满足：

- `index.html` 中 grep 找不到 `if(r.ok)`
- `index.html` 中 grep 找不到 `r.error`
- `index.html` 中已完成 `r.songs -> r.items`
- `history.html` 中 grep 找不到 `if(r.error)`
- 两个页面均已包含 `try/catch` 错误处理

## 4. 备注

1. 两个页面均已接入现有 `/js/api.js`，由 `unwrap` 统一处理 `Result` 包裹
2. `history.html` 的任务列表读取保持为 `r.tasks`，符合本次实现要求
3. `index.html` 中轮询失败时，当前行为为停止轮询并在状态区显示错误信息，避免持续静默失败
