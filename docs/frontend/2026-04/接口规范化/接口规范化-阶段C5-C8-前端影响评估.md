# 接口规范化-阶段C5-C8-前端影响评估

## 1. 目标与范围

本文用于评估阶段 C5-C8 中以下 controller 的前端影响范围：

- `sources_controller`
- `music_controller`
- `files_controller`
- `home_controller`
- `health_controller`

本次 **只做前端影响评估，不进入实现**。

评估依据：

- `backend/controllers/*.py` 当前实现
- `frontend/` 目录中的真实页面与接口调用
- 现有首页、歌曲页、audio-sources 页的实际依赖情况

---

## 2. 结论总览

| Controller | 是否有对应前端页面 | 结论 |
|-----------|------------------|------|
| `sources_controller` | 无明确对应页面 | 无前端改动 |
| `music_controller` | 无明确对应页面 | 无前端改动 |
| `files_controller` | 无对应前端页面 | 无前端改动 |
| `home_controller` | 无独立前端页面（仅保留文件路由） | 无前端改动 |
| `health_controller` | 无对应前端页面 | 无前端改动 |

> 说明：上述结论指“本阶段不存在需要跟随 controller 规范化而改造的专属前端页面”。个别接口虽然在业务上可能被未来页面复用，但在当前代码库中并没有直接页面接入，因此当前可判定为 **无前端改动**。

---

## 3. 分项评估

### 3.1 `sources_controller`

#### 3.1.1 后端接口范围

当前 `sources_controller` 提供：

- `GET /api/sources`
- `PUT /api/sources/switch`
- `GET /api/sources/search`

主要用途是：

- 获取可用音乐源列表
- 切换当前激活的音乐源
- 基于当前音乐源搜索音乐

#### 3.1.2 前端页面排查结果

在当前 `frontend/` 目录中，未发现页面直接调用：

- `/api/sources`
- `/api/sources/switch`
- `/api/sources/search`

也未发现与“音乐源切换 / 搜索提供方”明确对应的页面入口。

当前前端中的“音源管理”页面 `frontend/pages/audio-sources/*` 对应的是：

- `audio_sources_controller`

而不是 `sources_controller`。

两者职责不同：

- `audio_sources_controller`：管理上传到系统内的音频资源
- `sources_controller`：管理外部音乐提供方（如 Spotify / local_file）

#### 3.1.3 结论

**无前端改动。**

当前没有对应前端页面，也没有页面直接依赖其返回结构，因此阶段 C5 若只做 `sources_controller` 的接口规范化，不会触发现有前端页面改造。

#### 3.1.4 备注

若未来新增“音乐源切换 / 外部搜索”页面，则需要新开前端方案，不属于本次 C5-C8 范围。

---

### 3.2 `music_controller`

#### 3.2.1 后端接口范围

当前 `music_controller` 提供：

- `POST /api/music/upload`
- `GET /api/music/download/midi/<filename>`

其中：

- `/upload` 用于上传音乐文件到 OSS
- `/download/midi/<filename>` 用于下载生成的 MIDI 文件

#### 3.2.2 前端页面排查结果

在当前 `frontend/` 目录中，未发现页面直接调用：

- `/api/music/upload`
- `/api/music/download/midi/...`

当前项目中的歌曲创建页面 `frontend/pages/songs/add.html` 已切换为：

- 选择既有 `audio_source_id`
- 不再直接上传音乐文件

页面文案中已有提示：

- “请选择已存在的音源 ID，文件上传已下线。”

这说明当前前端主流程已经不再依赖 `music_controller.upload`。

同时，当前也未发现专门的“下载 MIDI”前端按钮或页面直接接入 `/api/music/download/midi/<filename>`。

#### 3.2.3 结论

**无前端改动。**

当前没有对应前端页面，也没有现有页面直接依赖 `music_controller` 的返回结构。

#### 3.2.4 备注

如果未来在 transcribe 结果页加入“下载 MIDI”按钮，届时才需要单独评估该接口在前端的调用方式与错误处理；当前阶段无需改造。

---

### 3.3 `files_controller`

#### 3.3.1 后端接口范围

当前 `files_controller` 提供：

- `GET /api/files/oss-url`

用于根据：

- `song_id`
- `session_id`
- `audio_source_id`

查询对应文件的 OSS URL。

支持的 `type` 包括：

- `audio`
- `melody`
- `chord`
- `recording`
- `source`

#### 3.3.2 前端页面排查结果

在当前 `frontend/` 目录中，未发现页面直接调用：

- `/api/files/oss-url`

当前各页面对音频资源的处理仍以直接读取已有字段为主，例如：

- songs 相关页面直接读取 `audio_path / melody_path / chord_path`
- recordings 相关页面直接读取 `file_path`
- audio-sources 详情页直接读取 `file_path`

因此 `files_controller` 当前属于“后端提供了统一文件解析能力，但前端尚未接入”的状态。

#### 3.3.3 结论

**无前端改动。**

当前没有对应前端页面，也没有现有页面依赖该接口。

#### 3.3.4 备注

这类接口更像公共能力接口。若后续前端统一把“文件路径 → 可访问 URL”的解析逻辑从页面剥离出来，则可能会引入前端改造；但这属于新一轮优化，不是本阶段必须动作。

---

### 3.4 `home_controller` + `health_controller`

#### 3.4.1 `home_controller` 现状

当前 `home_controller` 实际只保留：

- `GET /outputs/<path:filename>`

其文件头注释也明确写明：

- “前端页面路由 - 已移交给 Nginx”

这说明它已经不是一个承载页面渲染逻辑的 controller，而只是一个遗留/补充的文件服务路由。

#### 3.4.2 `health_controller` 现状

当前 `health_controller` 提供：

- `GET /api/health`
- `GET /api/status`
- `POST /api/db/test`

这些接口主要用于：

- 健康检查
- 系统状态查询
- 数据库连接测试

#### 3.4.3 前端页面排查结果

在当前 `frontend/` 目录中，未发现页面直接调用：

- `/api/health`
- `/api/status`
- `/api/db/test`
- `/outputs/...`

也未发现对应的“系统状态页 / 健康检查页 / 运维页”。

首页 `frontend/index.html` 虽然是前端主页，但它当前读取的是：

- `/api/songs/list?limit=1`
- `/api/artists/list?limit=1`
- `/api/audio-sources/list?limit=1`

并不依赖 `home_controller` 或 `health_controller`。

#### 3.4.4 结论

**无前端改动。**

- `home_controller`：无独立前端页面，对当前前端页面无改造影响
- `health_controller`：无对应前端页面，对当前前端页面无改造影响

#### 3.4.5 备注

若未来新增后台运维页面（展示服务状态、数据库状态、版本信息等），才需要基于 `health_controller` 新开前端方案；当前不在本次范围内。

## 4. 风险与边界说明

### 4.1 本文的“无前端改动”定义

本文中“无前端改动”指的是：

- 当前代码库中没有对应页面
- 当前页面没有直接调用该 controller 的接口
- 因此本阶段不需要为了接口规范化去修改既有页面

这不等于这些 controller 永远不需要前端接入，而是说明：

- **在当前项目现状下，它们不构成本轮前端改造任务。**

### 4.2 与其他阶段的边界

需要特别区分：

1. `audio_sources_controller`
   - 有真实前端页面
   - 已在 C4 单独出页面改造方案
2. `sources_controller`
   - 无现有页面接入
   - 本文判定为无前端改动

两者名称相近，但职责不同，不能混淆。

### 4.3 旁路影响说明

虽然 C5-C8 本文结论都是“无前端改动”，但仍建议在联调时留意以下旁路场景：

- 若后端顺手改动了与首页统计、歌曲创建、扒谱下载相关的旧接口语义，可能会间接影响已有页面
- 这类影响不来自 `sources/music/files/home/health` 的“专属前端页面”，而来自跨模块共享接口的副作用

当前代码审查范围内，尚未发现这类直接风险，但联调时仍建议做一次基础回归。

## 5. 验收结论

### 5.1 本阶段前端任务判断

- `sources_controller`：无前端改动
- `music_controller`：无前端改动
- `files_controller`：无前端改动
- `home_controller`：无前端改动
- `health_controller`：无前端改动

### 5.2 建议的联调回归项

虽然无需页面改造，但联调时建议最低限度确认：

- [ ] 首页仍可正常打开，不受 C5-C8 后端改动影响
- [ ] 歌曲创建页仍不依赖 `/api/music/upload`
- [ ] 现有页面中没有遗漏的 `/api/files/oss-url` 接入点
- [ ] Nginx 页面路由仍正常，未误依赖 `home_controller`

---

## 6. 结论

阶段 C5-C8 本次评估涉及的 `sources_controller`、`music_controller`、`files_controller`、`home_controller`、`health_controller`，在当前项目中都 **没有需要跟随改造的对应前端页面**，因此可统一结论为：

**无前端改动。**

本轮剩余前端设计重点应继续放在：

- C4 `audio_sources_controller` 对应页面改造

而 C5-C8 这几个 controller 在当前前端侧仅需保留“已评估、暂无页面改造需求”的记录即可。
