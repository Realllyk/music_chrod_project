# 旋律提取升级方案

## 阶段 1：后端需求分析

### 1.1 需求目标
将当前“直接对整首流行歌做旋律提取”的方案，升级为更适合有人声流行曲的主旋律提取流程。

核心目标：
- 提高流行歌主旋律提取的准确度
- 优先提取主唱旋律
- 降低伴奏、鼓、贝斯、和声对结果的干扰
- 输出仍保持为 MIDI，方便后续使用

### 1.2 涉及的后端模块

| 模块 | 操作类型 | 说明 |
|------|---------|------|
| controller/transcribe_controller.py | 修改 | 调整任务流程，增加“预处理/人声分离”步骤 |
| service/转录任务编排层 | 新增 / 修改 | 建议把“下载音频 → 分离 → 提取 → 上传结果”拆成更清晰的流程 |
| transcriber/librosa/melody.py | 修改 | 继续复用现有旋律提取，但输入改为“人声轨”而不是整首混音 |
| transcriber/spleeter/ 或 demucs/ | 新增 / 完善 | 引入或完善“人声分离”能力 |
| utils/aliyun_oss.py | 复用 | 继续负责下载原音频和上传结果 |
| 数据库存储（songs / transcribe_tasks） | 轻度修改可选 | 是否记录中间产物路径，如 vocal stem 路径，需要确认 |
| 日志模块 | 修改 | 增加“当前执行到哪一步”的日志，方便排查效果差的问题 |

只是后端内部默认改成：
- 先做人声分离
- 再做旋律提取

#### 增强方案
可在接口中增加可选参数，例如：
- `target=vocal / instrumental`

### 1.7 风险点 / 依赖项 / 待确认项

```text
🚨 风险：
  - 人声分离会增加处理耗时，任务比现在更慢
  - 若分离算法依赖较重（如 demucs/spleeter），部署环境需要补依赖
  - 某些歌曲的人声与伴奏混合严重，即使先分离，也不一定完全准确
  - 如果当前服务器资源有限，长音频处理可能占用较高 CPU / 内存

🔗 依赖：
  - 需要明确优先使用哪套人声分离方案（demucs / spleeter / 其他）
  - 需要确认当前运行环境是否已具备相关模型和依赖
  - 需要确认是否允许保存中间产物（vocal stem）到 OSS

❓ 待确认：
  - 旋律提取的目标是否明确为“主唱主旋律”
  - 是否只考虑中文/英文流行歌，还是也要兼容现场录音、翻唱、混剪
  - 是否需要让用户下载/试听分离出来的人声轨
```

## 阶段 3：API 契约设计

建议分成 V1 最小可行方案 和 V2 扩展方案。

### 3.1 API：启动旋律提取任务（V1，推荐先做）
- **接口类型：** 已有接口修改（复用现有 `/api/transcribe/start`，调整后端内部处理流程，不新增接口）
- **用途：** 发起歌曲主旋律提取
- **路径：** `/api/transcribe/start`
- **方法：** `POST`
- **鉴权：** 不需要

**请求参数：**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| song_id | int | 是 | 歌曲 ID |
| mode | string | 是 | 固定为 `melody` |

**请求示例：**
```json
POST /api/transcribe/start
Content-Type: application/json

{
  "song_id": 123,
  "mode": "melody"
}
```

**后端内部默认行为：**
1. 下载原始音频
2. 先做人声分离
3. 取人声轨做人声主旋律提取
4. 生成 MIDI
5. 上传结果
6. 更新任务状态

**成功响应（200）：**

| 字段 | 类型 | 说明 |
|------|------|------|
| ok | bool | 是否提交成功 |
| task_id | string | 任务 ID |
| message | string | 描述信息 |

**响应示例：**
```json
{
  "ok": true,
  "task_id": "task_xxx",
  "message": "Task submitted"
}
```

**错误响应：**

| HTTP状态码 | 说明 |
|-----------|------|
| 400 | 参数错误 |
| 404 | 歌曲不存在 |
| 500 | 任务创建失败 |

### 3.2 API：查询任务状态（建议增强）
- **接口类型：** 已有接口增强（复用现有 `/api/transcribe/status/{task_id}`，建议增强返回信息，不新增接口）
- **用途：** 查看旋律提取执行到哪一步
- **路径：** `/api/transcribe/status/{task_id}`
- **方法：** `GET`
- **鉴权：** 不需要

建议增加的状态阶段：
当前只有：
- `pending`
- `processing`
- `completed`
- `failed`

建议增强为更细的“步骤状态”，例如放到任务详情字段里：

| 字段 | 类型 | 说明 |
|------|------|------|
| status | string | 总状态 |
| stage | string | 当前阶段 |
| progress_message | string | 文字说明 |

#### stage 建议值
- `downloading_audio`
- `separating_vocals`
- `extracting_melody`
- `generating_midi`
- `uploading_result`
- `completed`
- `failed`

**响应示例：**
```json
{
  "task_id": "task_xxx",
  "status": "processing",
  "stage": "separating_vocals",
  "progress_message": "正在分离人声",
  "result_path": null,
  "error": null
}
```

> 注：这一项会涉及任务表字段是否扩展；如果先尽量少改，可先只打日志，不改接口。

## 阶段 4：数据交互说明

### 4.1 当前流程（问题所在）

```text
用户点击“旋律提取”
   │
   ▼
前端 → POST /api/transcribe/start
   │
   ▼
后端 Controller
   │
   ▼
下载原始歌曲（整首混音）
   │
   ▼
直接做旋律提取
   │
   ▼
生成 MIDI
   │
   ▼
上传结果
```

当前问题：
因为输入是整首混音，所以旋律提取器会同时听到：
- 主唱
- 钢琴
- 吉他
- 鼓
- 贝斯
- 和声

结果就是：
- 很难稳定识别“主唱那条线”
- 旋律容易漂、错、碎

### 4.2 升级后流程（推荐）

```text
用户点击“旋律提取”
   │
   ▼
前端 → POST /api/transcribe/start
   │
   ▼
后端入口层：Controller
   │   参数校验、创建任务
   ▼
业务层：任务编排
   │
   ├─ Step 1: 下载原始音频
   │
   ├─ Step 2: 人声分离
   │        输入：整首歌曲
   │        输出：人声轨 vocal stem
   │
   ├─ Step 3: 对人声轨做旋律提取
   │        输出：音高序列 / 音符序列
   │
   ├─ Step 4: 生成 MIDI
   │
   ├─ Step 5: 上传 MIDI 到 OSS
   │
   └─ Step 6: 更新 songs / tasks 状态
   ▼
响应链路：任务状态查询 → 前端查看进度/结果
```

### 4.3 为什么要先做人声分离
因为目标是“主唱唱的旋律”，那最合理的输入就应尽量接近“只有主唱的声音”。
这样系统任务更单纯，不需要在一堆伴奏中猜“想听谁”。

### 4.4 中间产物处理策略

#### 方案 A：临时文件，不落库，不上传
流程：
- 下载原音频
- 生成 vocal stem
- 提取完成后删除临时文件

**优点：**
- 改动小
- 不污染数据库
- OSS 不增加中间文件

**缺点：**
- 不能回看分离效果
- 重试时需要重新分离

#### 方案 B：保存 vocal stem 到 OSS
流程：
- 分离后上传 vocal stem
- 记录路径
- 后续提取/调试可复用

**优点：**
- 方便调试
- 可试听分离质量
- 后续重复提取可复用

**缺点：**
- 增加存储成本
- 需要考虑数据库字段或额外记录

当前建议：第一阶段先用方案 A，先验证效果提升，再考虑持久化中间产物。

## 方案设计细化

### A. 升级目标拆分

#### 目标 1：先把输入变干净
不是直接在整首混音上提旋律，而是先尽量提取出人声。

#### 目标 2：保留现有旋律提取器
当前 `librosa` 旋律提取可先继续保留，不急着全部推翻。
先把输入换成“人声轨”，通常效果会比现在稳定很多。

#### 目标 3：先做最小闭环
第一版建议只做：
- 原曲下载
- 人声分离
- 人声旋律提取
- MIDI 输出

先把链路跑顺。

### B. 算法/模块建议

#### B1. 人声分离层

| 方案 | 建议级别 | 说明 |
|------|---------|------|
| Demucs | 高 | 对流行歌曲、人声分离通常更稳，更适合当前场景 |
| Spleeter | 中 | 上手较快，但部分歌曲分离质量可能不如 Demucs |
| 继续不用分离 | 低 | 不建议，当前效果已验证不理想 |

建议优先 Demucs。
原因：
- 当前歌曲大多为流行曲、带人声
- 目标是提取主唱旋律
- 该场景下，人声分离质量比后续参数微调更重要

#### B2. 旋律提取层
第一阶段建议：
- 继续使用现有 `librosa` 提取器
- 但输入改为人声轨

优点：
- 改动相对可控
- 可先验证“先分离再提取”方向是否明显提升效果

如后续仍不理想，再考虑替换或增强旋律提取算法。

### C. 日志与可观测性建议
建议至少增加以下日志：

```text
[task_id=xxx] 开始下载原始音频
[task_id=xxx] 原始音频下载完成
[task_id=xxx] 开始人声分离
[task_id=xxx] 人声分离完成，输出文件: xxx.wav
[task_id=xxx] 开始旋律提取
[task_id=xxx] 旋律提取完成，识别音符数: xxx
[task_id=xxx] 开始生成 MIDI
[task_id=xxx] MIDI 已上传: xxx
```

这样可以快速定位卡在下载、分离、提取、生成还是上传阶段。

### D. 部署与资源建议
由于新流程会引入“人声分离”，需提前评估：

#### 资源影响
- CPU 使用升高
- 内存占用升高
- 单任务耗时增加

#### 风险控制建议
第一版先限制：
- 单文件大小
- 单次并发任务数
- 超时时间

避免多首歌同时处理导致资源耗尽。

## 建议实施顺序

### 第一阶段（推荐先做）
#### 最小可行升级版
1. 保留现有 `/api/transcribe/start`
2. 当 `mode=melody` 时：
   - 下载原始音频
   - 先做人声分离
   - 用分离后的人声做旋律提取
   - 输出 MIDI
3. 不改数据库
4. 中间人声文件只做临时文件
5. 增加详细日志

这一版的目标不是一次做到专业级扒谱，而是先让“流行歌主旋律提取效果比现在明显更像主唱在唱的线”。

### 第二阶段（后续可做）
1. 增加任务进度细分状态
2. 可选保存 vocal stem 到 OSS
3. 增加重试机制
4. 优化旋律提取参数
5. 针对男女声、高低音域做自适应范围调整

### 第三阶段（如果继续深入）
1. 提升人声主旋律精度
2. 处理和声/双人声场景
3. 更自然地切分音符
4. 做试听对比和结果可视化

## 后端视角最终建议
当前旋律提取效果差，主要问题不是参数没调好，而是输入给错了。
因此第一步不应先猛调现有算法，而应先把流程改成：

**整首歌 → 人声分离 → 人声旋律提取**

这才符合当前“流行歌、有人声”为主的数据特征。
