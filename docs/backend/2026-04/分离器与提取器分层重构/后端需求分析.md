# 分离器与提取器分层重构 - 后端需求分析

> 面向 `music_chrod_project` 后端的分层重构分析文档。
> 本文档仅用于需求分析与设计确认，**不代表已进入实现阶段**。

---

## 阶段 1：后端需求分析

### 1.1 需求目标
本次需求的后端目标，是将当前转录链路中“音轨分离（Demucs / Spleeter）”与“旋律/和弦提取（Librosa / Basic Pitch 等）”的耦合实现拆开，重构为更清晰的分层结构，使系统具备以下能力：

1. **职责分离**：让分离器只负责 stem 分离，让提取器只负责 melody/chord 提取。
2. **组合扩展**：支持“任意分离器 + 任意提取器”的可组合能力，避免继续膨胀出大量硬编码类。
3. **分层清晰**：将当前 `transcribe_controller.py` 中混杂的控制器逻辑、业务逻辑、数据库访问、异步调度、上传流程拆回到 Controller / Service / Mapper / Pipeline / Separator 各层。
4. **兼容现有调用方**：保持对外 API 路径、请求参数、响应结构、`config.json` 字段名与既有语义尽量不变，使前端与已有调用方无感。
5. **降低后续维护成本**：为后续继续接入新的分离器、优化和弦算法、优化 melody 后处理提供更稳定的骨架。

简言之，这不是对外功能新增，而是一次**后端内部架构重整 + 职责归位**。

### 1.6 API 兼容性影响
当前设计目标是**不影响已有接口调用方**。

需要重点确保以下兼容项：

1. `POST /api/transcribe/start` 仍返回任务启动结果，字段含义不变。
2. `GET /api/transcribe/status/<task_id>` 返回的任务状态结构不变。
3. `GET /api/transcribe/song/<song_id>` 返回的任务列表结构不变。
4. melody 链路输出中的以下字段语义不变：
   - `notes`
   - `total_notes`
   - `duration_sec`
   - `vocals_path`
5. chord 链路输出中的以下字段语义不变：
   - `chords`（或既有兼容字段）
   - `duration`
6. `config.json` 中以下字段继续生效：
   - `transcription.vocal_separation.provider`
   - `transcription.vocal_separation.fallback_order`
   - `transcription.melody.backend`
   - `transcription.algorithm.chord`

**后端视角建议：**
虽然本次目标是不改 API，但在实现与回归阶段必须把“接口无变化”当成明确验收项，而不是默认成立。

### 1.7 风险点 / 依赖项 / 待确认项

```text
🚨 风险：
  - 风险1：本次属于内部架构重构，涉及链路长、模块多，若切换顺序不当，容易导致转录链路整体不可用。
  - 风险2：Demucs / Spleeter 现有逻辑拆出后，临时目录清理、preserved_vocals_path 保留、MIDI 保存时序等资源管理细节容易回归。

  - 是否需要在本次重构中顺带统一 chord 返回字段的兼容行为（目前文档倾向“不处理”，后续单开需求）。
  - 是否需要补最小化的端到端自动化回归脚本，还是以人工回归为主。
  - 旧 `transcriber/demucs/`、`transcriber/spleeter/` 目录的删除时机是否严格放在最后一步回归通过后执行。
```

## 阶段 3：API 契约设计

> 本次需求目标为“对外契约不变，内部职责重构”。因此本节以接口兼容说明为主。

### API：启动转录任务
- **接口类型：** 复用无改动
- **用途：** 创建一条 melody/chord 转录任务，并异步启动后台处理
- **路径：** `/api/transcribe/start`
- **方法：** `POST`
- **鉴权：** 不需要

**请求参数：**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `song_id` | int | 是 | 歌曲 ID |
| `mode` | string | 是 | 提取模式，取值为 `melody` 或 `chord` |

**请求示例：**
```json
POST /api/transcribe/start
Content-Type: application/json

{
  "song_id": 12,
  "mode": "melody"
}
```

**成功响应（200）：**
| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | int | 0 表示成功 |
| `message` | string | 描述信息 |
| `data.task_id` | string | 新建任务 ID |
| `data.status` | string | 初始状态，通常为 `pending` |

**兼容性要求：**
- 接口路径不变
- `mode` 取值语义不变
- 响应结构不变
- 仍采用“先返回 task_id，再后台异步执行”的模式

### API：查询任务状态
- **接口类型：** 复用无改动
- **用途：** 根据任务 ID 查询转录任务当前状态与结果信息
- **路径：** `/api/transcribe/status/<task_id>`
- **方法：** `GET`
- **鉴权：** 不需要

**请求参数：**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `task_id` | string | 是 | 任务 ID，路径参数 |

**成功响应（200）：**
| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | int | 0 表示成功 |
| `message` | string | 描述信息 |
| `data.task_id` | string | 任务 ID |
| `data.song_id` | int | 对应歌曲 ID |
| `data.mode` | string | 任务模式 |
| `data.status` | string | 当前状态 |
| `data.result_path` | string/null | 成功时的结果路径 |
| `data.error` | string/null | 失败时的错误信息 |

**兼容性要求：**
- 返回字段名与原接口保持一致
- 任务状态流转语义不变：`pending → processing → completed / failed`

### API：按歌曲查询任务列表
- **接口类型：** 复用无改动
- **用途：** 查询指定歌曲关联的转录任务列表
- **路径：** `/api/transcribe/song/<song_id>`
- **方法：** `GET`
- **鉴权：** 不需要

**请求参数：**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| `song_id` | int | 是 | 歌曲 ID，路径参数 |

**成功响应（200）：**
| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | int | 0 表示成功 |
| `message` | string | 描述信息 |
| `data.song_id` | int | 歌曲 ID |
| `data.tasks` | array | 任务列表 |

**兼容性要求：**
- 返回体顶层结构不变
- 任务列表字段保持兼容
- 排序语义保持与当前实现一致

## 阶段 4：数据交互说明

### 4.1 完整请求-响应链路

#### Melody 链路

```text
用户发起 /api/transcribe/start
   │
   ▼
Controller
  - DTO 校验
  - 调用 TranscribeService.start(song_id, mode)
   │
   ▼
TranscribeService.start
  - 校验 song 是否存在
  - 通过 TranscribeTasksMapper.insert 创建任务
  - 启动后台线程 run_transcription(task_id, song_id, 'melody')
   │
   ▼
TranscribeService.run_transcription
  - 更新任务状态 processing
  - 下载原始音频
  - 通过 pipelines.create_melody_pipeline() 组装 pipeline
   │
   ▼
MelodyPipeline
  - separator.separate(audio_path, {'vocals'})
  - 保留 vocals 文件到可持久路径
  - transcriber.extract_melody(vocals_path)
   │
   ▼
TranscribeService.run_transcription
  - pipeline.save_midi(...)
  - 上传 MIDI 到 OSS
  - 上传 vocals 到 OSS
  - 回写歌曲结果与 melody analysis
  - 更新任务状态 completed / failed
   │
   ▼
用户再调 /api/transcribe/status/<task_id>
   │
   ▼
Controller → TranscribeService.get_task_status → TranscribeTasksMapper.find_by_task_id
   │
   ▼
返回统一响应
```

#### Chord 链路

```text
用户发起 /api/transcribe/start
   │
   ▼
Controller
  - DTO 校验
  - 调用 TranscribeService.start(song_id, mode)
   │
   ▼
TranscribeService.run_transcription(..., 'chord')
  - 下载原始音频
  - 通过 pipelines.create_chord_pipeline() 组装 pipeline
   │
   ▼
ChordPipeline
  - separator.separate(audio_path, {'vocals','drums','bass','other'})
  - 内部将 bass + other + drums 合成为 accompaniment
  - transcriber.extract_chords(accompaniment_path)
   │
   ▼
TranscribeService.run_transcription
  - pipeline.save_midi(...)
  - 上传 MIDI
  - 更新歌曲结果与任务状态
   │
   ▼
返回统一响应
```

### 4.2 分层职责边界

| 层级 | 职责 | 本次要求 |
|------|------|----------|
| Controller | 入参校验、调用 Service、返回统一响应 | 不再直接组装 transcriber / pipeline，不再写 SQL |
| Service | 业务编排、异步任务、OSS、状态更新、分析缓存 | 成为转录主流程唯一编排入口 |
| Pipeline | 编排“分离 → 提取 → 资源清理”链路 | 不直接做 HTTP 和 DB |
| Separator | 只负责分离 stem | 不负责 melody / chord 分析 |
| Transcriber | 只负责提取 melody / chord 与保存 MIDI | 不负责分离与上传 |
| Mapper | 只负责 `transcribe_tasks` 表 CRUD | 不承载业务判断 |

### 4.3 本次链路重构的关键收益

1. **控制器瘦身**：避免 controller 继续成为业务、SQL、线程、外部依赖的聚合点。
2. **分离/提取解耦**：后续若需要 `demucs + basic_pitch`、`spleeter + librosa` 等组合，不必新增伪装类。
3. **降级链更合理**：从“切换一体化 transcriber”改为“切换 separator”，更符合配置语义。
4. **维护面收缩**：和弦分析重复逻辑可集中，后续优化只需改一处。

## 后端视角建议

### 建议 1：实现时按“先加后切、最后删”的顺序推进
这类重构不应直接原地大改，建议顺序为：
1. 先新增 separator / pipeline / mapper / service
2. 再切 controller 到新链路
3. 回归通过后最后删除旧 transcriber 目录

这样回退成本最低。

### 建议 2：把“接口不变”作为强验收项
虽然本次不是接口需求，但对外契约兼容就是本次成败关键之一。建议在实现阶段保留同一音频改造前后输出对比。

### 建议 3：日志必须能看见实际 pipeline 组合
例如：
- `demucs + basic_pitch`
- `spleeter + librosa`
- `passthrough + librosa`

否则排查降级行为会很困难。
