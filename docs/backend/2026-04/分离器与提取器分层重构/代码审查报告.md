## 1. Separator 层

### 问题 1-1：DemucsSeparator 空集合判断逻辑错误
**文件**：`separator/demucs.py`
**位置**：`separate` 方法第 1 行
**问题**：
```python
if not stems or not stems.issubset(self.SUPPORTED_STEMS):
```
`stems` 是 `FrozenSet[str]`，永远为 truthy，即使空集合 `frozenset()` 也如此。旧实现里 `stems` 可能为空（调用方传空集合），但这行永远走不到 `not stems` 分支。
**影响**：空集合请求不会触发错误，也不会正常返回，但实际调用路径中暂无此场景。
**建议**：删除 `not stems or`，仅保留 `not stems.issubset(...)`；或者如果设计上不允许空集，直接 `if not stems.issubset(...)` 即可。

### 问题 1-3：PassthroughSeparator.work_dir 空目录无实际用途
**文件**：`separator/passthrough.py`
**位置**：`separate` 方法
**问题**：当前实现创建了一个空目录作为 work_dir，让 Pipeline 的 `shutil.rmtree` 行为保持一致。但这个空目录不会被任何地方使用，且 `shutil.rmtree` 本身就能处理不存在的路径（`ignore_errors=True`），因此这个临时目录是多余的。
**影响**：轻微——多了不必要的 temp 文件创建逻辑。
**建议**：可接受当前实现（语义更清晰），也可以直接把 `work_dir` 设为空字符串 `''`，由 Pipeline 的 `ignore_errors=True` 兜底。

### 问题 1-4：DemucsSeparator 2stems 判断逻辑与旧实现不完全一致
**文件**：`separator/demucs.py`
**位置**：`separate` 方法
**问题**：当前判断为：
```python
is_two_stems = stems.issubset({'vocals', 'accompaniment'})
```
但旧 `DemucsMelodyTranscriber` 里只请求 `{'vocals'}`，而 `DemucsChordTranscriber` 请求 `{'vocals', 'drums', 'bass', 'other'}`（4stems）。`{'vocals', 'accompaniment'}` 这个集合在旧实现里从未实际出现（2stems 模式只有 vocals 请求）。
**影响**：如果未来某处传入 `{'vocals', 'accompaniment'}` 请求，会走 2stems 模式，但旧实现从未在此场景下运行过，可能存在回归风险。
**建议**：当前行为与旧 demucs 调用语义一致（`--two-stems vocals` 只请求 vocals），可接受；如果要完全兼容 `accompaniment` 请求，需额外测试验证。

## 2. Pipeline 层

### 问题 2-1：ChordPipeline.sample_rate 与 transcriber 采样率可能不一致
**文件**：`pipelines/chord_pipeline.py`
**位置**：`__init__` 方法
**问题**：
```python
def __init__(self, separator: VocalSeparatorBase, transcriber: ChordTranscriberBase, sr: int = 22050):
    self.sample_rate = sr
```
ChordPipeline 硬编码 `sr=22050`，而 `LibrosaChordTranscriber` 也有自己的 `self.sr = sr`（默认 22050）。两者目前碰巧相同，但：
- `ChordPipeline._mix_accompaniment` 用的是 `self.sample_rate`（来自 Pipeline）
- `transcriber.extract_chords` 用的是 transcriber 自己的 sr
如果 transcriber 的 sr 被配置为其他值（例如 44100），两者会产生采样率不一致，导致伴奏混合后的音频与 transcriber 期望的采样率不匹配。
**影响**：中等——如果 transcriber sr 配置改变，伴奏混合音频和 chord 提取的采样率会不一致，可能导致 chord 输出异常。
**建议**：在 Pipeline 构造时，从 transcriber 对象读取 `transcriber.sr` 作为 `self.sample_rate`，而不是硬编码。

---

### 问题 2-2：ChordPipeline 伴奏混合缺少 drums 轨的原始处理（与旧实现差异）
**文件**：`pipelines/chord_pipeline.py`
**位置**：`_mix_accompaniment` 方法
**问题**：旧 `DemucsChordTranscriber._analyze_chords` 混合顺序是 `bass + other`（不含 drums），代码中写的是：
```python
for name in ['bass', 'other']:
    y_track, _ = librosa.load(track_paths[name], sr=self.sample_rate)
    if harmonic is None:
        harmonic = y_track
    else:
        min_len = min(len(harmonic), len(y_track))
        harmonic = harmonic[:min_len] + y_track[:min_len]
```
而新的 `ChordPipeline` 混合顺序是 `bass + other + drums`（三个都加）。
```python
ACCOMPANIMENT_STEMS = ('bass', 'other', 'drums')
```
**影响**：高——chord 分析依赖的是非人声伴奏，而 drums 的频谱特征与 bass/other 完全不同，混入 drums 会改变 chroma 分析结果，可能导致和弦识别准确率下降或与旧实现完全不兼容。
**建议**：恢复为 `('bass', 'other')`，与旧实现保持一致。drums 不应参与和弦分析。

---

### 问题 2-3：Pipeline 工厂函数每次调用都重新创建 Transcriber 实例
**文件**：`pipelines/__init__.py`
**位置**：`create_melody_pipeline`、`create_chord_pipeline`
**问题**：
```python
def create_melody_pipeline() -> MelodyPipeline:
    transcriber = _create_melody_transcriber()  # 每次都重新构造
```
每次调用都 `from transcriber.xxx import XxxTranscriber` 并 `XxxTranscriber()`。而 `TranscribeService.run_transcription` 里每次任务都调一次 `create_melody_pipeline()`（via `TranscribeService.start`），所以每个任务都会重新构造 transcriber 实例。
**影响**：轻微——模型加载本身就有开销（demucs 模型大），这个行为与旧实现一致（旧实现里每次任务也是新建），所以不是回归问题。但从架构上说，transcriber 实例应该是可复用的（无状态）。
**建议**：可接受现状（与旧实现一致）；如果要优化，可以在 pipeline 内部缓存 transcriber 实例。

---

### 问题 2-4：Config 缓存变量命名与 config.json 字段不完全对应
**文件**：`pipelines/__init__.py`
**位置**：`_load_config` 函数
**问题**：缓存变量名叫 `_CONFIG_CACHE`，但 config.json 里实际字段是 `transcription.vocal_separation` / `transcription.melody` / `transcription.algorithm`。在 `_resolve_separator_order` 中访问路径是 `cfg.get('provider')`，这对应 `vocal_separation.provider`，是正确的。但 `_create_melody_transcriber` 里读的是 `melody.backend`，也是正确的。
**影响**：无问题，访问路径正确。

**问题**：

conn = get_db()

## 4. Service 层

### 问题 4-1：TranscribeService._save_and_upload_midi 中 midi_path 被 finally 清理后返回给调用方
**文件**：`services/transcribe_service.py`
**位置**：`finally` 块（第 123-127 行）
**问题**：
```python
finally:
    for path in (full_audio_path, midi_path, local_vocals_path):
        if path and os.path.exists(path):
            try:
                os.remove(path)  # ← 这里删了 midi_path
            except Exception:
                pass
```
而 `_save_and_upload_midi` 的返回值是 `(midi_path, result_path)`（第 115 行），`run_transcription` 里用 `midi_path` 做 `MelodyAnalysisService.build_melody_analysis(midi_path)` 的参数。这个调用发生在 `_save_and_upload_midi` 返回之后、`finally` 块执行之前，所以时序上是安全的：
```
midi_path, result_path = _save_and_upload_midi(...)  ← midi_path 在这里还有
analysis_payload = MelodyAnalysisService.build_melody_analysis(midi_path)  ← 在这里用了
finally: os.remove(midi_path)  ← 最后才删
```
**影响**：无问题。时序分析清楚。
**建议**：无需修改，但建议在 `finally` 注释里说明为什么 midi_path 在这里可以安全清理。

## 6. 整体链路

### 问题 6-1：MelodyAnalysisService.build_melody_analysis 依赖本地 MIDI 文件
**文件**：`services/transcribe_service.py`
**位置**：`run_transcription` 中的调用
**问题**：时序是：
1. `_save_and_upload_midi` 生成 midi_path 并上传 OSS
2. `MelodyAnalysisService.build_melody_analysis(midi_path)` 读本地 midi_path
3. `finally` 删 midi_path
这个顺序是正确的，因为 OSS 上传已经在第 1 步完成了。但如果在第 2 步 OSS 还没上传完就被删了怎么办？实际上 `_save_and_upload_midi` 是先写本地再上传，上传完成后才返回 midi_path，所以时序是安全的。
**影响**：无问题。
**建议**：无需修改，但建议在注释里说明这个时序依赖。

## 7. 潜在未覆盖场景

### 7-1：LibrosaChordTranscriber 的 HPSS 行为与 chord pipeline 语义错位
**文件**：`transcriber/librosa/chord.py`
**问题**：`LibrosaChordTranscriber.extract_chords` 内部自己做了 HPSS 分离，然后从 harmonic 提取多声部旋律。这个行为在旧的 `DemucsChordTranscriber` 里是用 demucs 做分离、再把 bass+other 混合后喂给 librosa 做 chroma 分析。现在新的 `ChordPipeline` 在 passthrough 模式下直接调 `transcriber.extract_chords(audio_path)`，librosa 会再做一次 HPSS——这在 passthrough 场景下是正确的（因为 passthrough 就是不分离、librosa 自己处理）。
但对于 demucs/spleeter 场景：Pipeline 先用 demucs 分离，再把 bass+other+drums 混合成 accompaniment，再调 `transcriber.extract_chords(accompaniment_path)`。此时 librosa 会再次对自己的输入做 HPSS，但输入已经是纯伴奏了，再次 HPSS 可能产生不同结果。
**影响**：中等——在 demucs/spleeter + librosa chord 场景下，chord 分析会经过两次 HPSS（一次在 demucs 分离，一次在 librosa 内部），可能导致 chord 输出与旧实现不一致。
**建议**：这实际上是旧实现的已有问题（librosa chord transcriber 自己会 HPSS），不是本次重构引入的。建议后续单独评估是否需要让 librosa chord transcriber 在外部已做分离的情况下跳过 HPSS。

---

### 7-2：BasicPitchMelodyTranscriber 的 `save_midi` 依赖 `self.notes` 状态
**文件**：`transcriber/basic_pitch/melody.py`
**问题**：`BasicPitchMelodyTranscriber.save_midi` 使用 `self.notes` 来构建 MIDI。而 `MelodyPipeline.extract_melody` 先调 `transcriber.extract_melody(vocals_path)`，再调 `pipeline.save_midi(output_path)`。由于 Pipeline 把 transcriber 存在 `self.transcriber` 里，所以 `save_midi` 能拿到 `self.transcriber.notes`。这与旧实现一致（DemucsMelodyTranscriber 也是把 delegate 存起来再调 save_midi）。
**影响**：无问题，行为与旧实现一致。
