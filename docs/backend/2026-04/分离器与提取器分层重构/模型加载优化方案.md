# 模型加载优化分析

> 生成时间：2026-04-24
> 关联需求：分离器与提取器分层重构
> 状态：待决策

---

## 1. 现状描述

### 1.1 当前每次任务都重新加载模型

在 `pipelines/__init__.py` 的工厂函数中，每次调用 `create_melody_pipeline()` / `create_chord_pipeline()` 都会：

```python
def create_melody_pipeline() -> MelodyPipeline:
    transcriber = _create_melody_transcriber()   # 每次都重新 import + 构造
    # ...
    separator = create_separator(provider)        # DemucsSeparator 每次都重新实例化
    # ...
    return MelodyPipeline(separator, transcriber)
```

而 `TranscribeService.run_transcription` 每处理一个任务都调一次工厂函数，因此每个任务都会：

- 重新加载 demucs 模型（htdemucs 约 1.4GB .pth 文件）
- 重新实例化 transcriber 对象（librosa / basic_pitch）

### 1.2 性能影响

| 场景 | 单次加载耗时 | 说明 |
|------|------------|------|
| DemucsSeparator（htdemucs） | ~5-15 秒 | 模型文件约 1.4GB，CPU 加载更慢 |
| SpleeterSeparator | ~1-3 秒 | TF SavedModel，约几百 MB |
| LibrosaMelodyTranscriber | < 1 秒 | 纯 Python 逻辑，无大模型 |
| BasicPitchMelodyTranscriber | < 1 秒 | onnxruntime 模型，约几十 MB |

主要瓶颈在 **demucs**。每次任务都等 demucs 模型加载完成才开始分离，5-15 秒的额外等待在端到端延迟中占比很大。

### 1.3 与旧实现的行为对比

旧实现里，每次 `run_transcription` 也会调 `_create_melody_transcriber()`，每次都重新 import + 构造。**这不是本次重构引入的新问题，是既有的架构选择。**

本次重构只是把"分离器+提取器"的耦合实现拆开了，模型加载模式本身没有变化。

## 2. 方案对比

### 方案 A：Session 级 Pipeline 缓存（推荐）

**思路**：在 pipelines 工厂层维护进程内缓存，同一个 provider+backend 组合只创建一次，后续直接复用。

**改动位置**：`pipelines/__init__.py`

**优点**：
- 改造成本最小，不涉及 separator/transcriber 内部
- 进程生命周期内模型只加载一次
- 对服务稳定性影响低（只改工厂层，不改业务逻辑）

**缺点**：
- 进程重启后仍需重新加载
- 多 worker（uwsgi/gunicorn）场景下每个 worker 独立加载，无法跨进程共享

**实现示意**：

```python
# pipelines/__init__.py
_pipeline_cache: Dict[str, MelodyPipeline] = {}

def create_melody_pipeline() -> MelodyPipeline:
    transcriber = _create_melody_transcriber()
    cache_key = _pipeline_cache_key('melody', _get_provider(), _get_backend())
    if cache_key in _pipeline_cache:
        return _pipeline_cache[cache_key]

    for provider in _resolve_separator_order():
        try:
            separator = create_separator(provider)
            pipeline = MelodyPipeline(separator, transcriber)
            _pipeline_cache[cache_key] = pipeline
            return pipeline
        except Exception as e:
            logger.warning(f'分离器初始化失败 provider={provider}: {e}')
    raise RuntimeError('No available separator')
```

**注意事项**：
- transcriber 实例通常是无状态的（不持有音频数据），可以安全复用
- separator 实例如果持有临时目录或模型引用，需要确保是线程安全的
- 需要预留手动刷新缓存的机制（config 变更时）

---

### 方案 B：Demucs Python API 直接实例化（绕过 CLI）

**思路**：不走 `demucs.separate.main()` CLI 路径，而是直接调用 `demucs.hdemucs.HDEMUCS` 的 Python 接口，自己管理模型生命周期。

**改动位置**：`separator/demucs.py`

**实现路径**：
```python
from demucs.pretrained import get_model
from demucs.hdemucs import HDemucs

class DemucsSeparator(VocalSeparatorBase):
    _model_cache: Dict[str, HDemucs] = {}

    @classmethod
    def _get_model(cls, model_name: str):
        if model_name not in cls._model_cache:
            model = get_model(model_name)
            model.eval()
            cls._model_cache[model_name] = model
        return cls._model_cache[model_name]
```

**优点**：
- 更细粒度的缓存控制
- 可以避免每次都走 CLI 的参数解析开销

**缺点**：
- 依赖 demucs 内部 API，不保证版本兼容性
- 需要自行处理模型推理逻辑（`model.forward(audio)`），而不是靠 CLI 输出文件
- 如果 demucs 升级改了接口，缓存逻辑可能失效

**风险提示**：
demucs 官方推荐的使用方式就是 CLI，不保证 Python API 的稳定性。这个方案有维护成本。

---

### 方案 C：独立 Demucs 推理服务

**思路**：启动一个独立的 `demucs-server` 进程，通过 HTTP 或 Unix Socket 提供 stem 分离服务。模型在服务启动时加载一次，所有请求共享。

**架构**：

```
Flask (transcribe_controller)
       ↓ HTTP 请求
demucs-server (独立进程)
       ↓ 模型推理
返回分离后的 stem 文件路径
```

**优点**：
- 模型只加载一次，多 worker 共享
- 可以独立扩展分离服务
- 对主服务无侵入

**缺点**：
- 引入服务间通信延迟（本地 socket 约 1-5ms）
- 需要额外的进程管理与健康检查
- 部署复杂度显著提升
- 如果 demucs-server 崩溃，影响所有转录任务

**适用场景**：生产环境、高并发、长期运行的服务。

## 3. 推荐结论

| 阶段 | 推荐方案 | 说明 |
|------|---------|------|
| 本次重构期间 | 方案 A（Session 级缓存） | 改动最小，不影响本次重构进度 |
| 后续优化（中期） | 方案 B（Python API 实例化） | 如果 demucs 内部 API 稳定，可以做 |
| 长期生产优化 | 方案 C（独立推理服务） | 需要单独立项评估 |

---

## 4. 待确认事项

在决定之前，需要确认：

1. **服务部署方式**：是长期运行的 Flask 进程（如 gunicorn/uwsgi），还是短生命周期场景（如 Serverless）？
   - 长期运行 → 方案 A 收益明显
   - 短生命周期 → 方案 A 收益有限

2. **并发规模**：每秒大概多少转录任务？如果并发量低（< 1 req/s），模型加载开销不是主要瓶颈。

3. **多 worker 情况**：是否用了 gunicorn `-w N` 或类似的多 worker 部署？每增加一个 worker，模型会多加载一次。

4. **对 demucs Python API 的态度**：
   - 如果愿意接受版本耦合风险 → 方案 B 可行
   - 如果希望保持对官方 CLI 的依赖 → 方案 B 保守，仅做方案 A
