# 旋律结果可视化方案 - 后端需求分析（第三版）

## 1. 修订背景

根据 Edward 最新审批意见，本版必须修正以下 3 点：

1. **仅针对本次新增接口，统一响应格式**
   ```json
   {
     "code": 200,
     "description": "success",
     "result": { }
   }
   ```
2. **返回内容不得平铺在顶层，必须全部收敛到 `result` 下**
3. **数据库方案调整**
   - `songs` 表新增：`melody_key`、`chord_key`
   - 分析缓存表要承担简谱数据缓存职责，避免每次查询都重新生成
   - 需先确认当前分析表结构，再决定新增/复用方式

## 2. 当前库表现状确认

根据现有数据库设计文档 `docs/plan/DATABASE_DESIGN.md`：

### 2.1 `songs` 表当前已有字段
- `id`
- `title`
- `artist_id`
- `category`
- `duration`
- `audio_path`
- `melody_path`
- `chord_path`
- `status`
- `source`

- `created_at`
- `updated_at`
- `deleted_at`

**结论：当前 `songs` 表尚未包含 `melody_key`、`chord_key`。**

### 2.2 当前分析表实际名称与字段
数据库设计文档中现有表名为：`song_analysis`

字段如下：
- `id`
- `song_id`
- `analysis_type`
- `result_json`
- `midi_path`
- `created_at`

**重要说明：**
- 审批意见中写的是 `songs_analysis`，但当前项目设计文档中实际存在的是 `song_analysis`
- 本次后端方案建议：**不新增第二张近似表，继续复用现有 `song_analysis` 作为审批意见中的分析缓存表**
- 若后续数据库中真实表名已经是 `songs_analysis`，则实现时按线上实际表名落地；但从当前文档分析角度，现阶段应以 **“复用现有分析缓存表，不重复造表”** 为原则

## 3. 本次后端改动范围

### 3.1 Controller 层

#### `backend/controllers/songs_controller.py`
新增只读接口：
- `GET /api/songs/<int:song_id>/melody-analysis`

职责：
- 校验歌曲是否存在
- 从分析缓存表读取 `melody` 类型的结果
- 返回统一包装后的 JSON
- **禁止在该接口内触发实时 MIDI 解析**

### 3.2 Service 层

#### `backend/services/songs_service.py`
新增/调整：
- `get_melody_analysis(song_id)`
  - 查询歌曲
  - 查询旋律分析缓存
  - 组装新接口响应格式
- `save_melody_analysis(song_id, analysis_payload, midi_path, melody_key)`
  - 在旋律转录成功后写入分析缓存
  - 同步回写 `songs.melody_key`

#### `backend/services/melody_analysis_service.py`（新建）
职责调整为：
- 负责在转录完成后，将 MIDI 解析为结构化旋律分析结果
- 产出可直接缓存到分析表的数据结构
- 不对外暴露“实时查询即解析”的能力

建议提供方法：
- `build_melody_analysis(midi_path) -> dict`

### 3.3 Mapper 层

#### `backend/mappers/song_analysis_mapper.py`
需要补齐：
- 按 `song_id + analysis_type` 查询分析结果
- 旋律分析缓存 upsert 能力
- `result_json` 的序列化/反序列化处理

#### `backend/mappers/songs_mapper.py`
需要补齐：
- 更新 `songs.melody_key`
- 预留未来更新 `songs.chord_key` 的能力

### 3.4 转录流程

#### `backend/controllers/transcribe_controller.py`
在 `mode == 'melody'` 的成功分支中补入：
1. 调用 `MelodyAnalysisService.build_melody_analysis(midi_path)`
2. 调用 `SongsService.save_melody_analysis(...)`
3. 保存分析缓存后，更新 `songs.melody_key`
4. 若缓存写入失败，不能默默吞掉，需要记录日志并按约定返回失败或标记任务异常

#### `backend/transcriber/librosa/melody.py`
继续保留在 MIDI 生成阶段写默认拍号 `4/4`：
```python
midi = pretty_midi.PrettyMIDI(initial_tempo=float(self.tempo))
midi.time_signature_changes.append(pretty_midi.TimeSignature(4, 4, 0))
```

## 4. 数据库变更方案（修订后）

## 4.1 `songs` 表新增字段

新增字段：
- `melody_key` VARCHAR(32) DEFAULT NULL COMMENT '旋律分析缓存key，绑定分析表记录key'
- `chord_key` VARCHAR(32) DEFAULT NULL COMMENT '和弦分析缓存key，绑定分析表记录key'

用途：
- `melody_key`：指向歌曲当前生效的旋律分析缓存
- `chord_key`：为未来和弦结果可视化或和弦缓存能力预留

说明：
- 这两个字段是**业务绑定键**，不是直接替代 `melody_path/chord_path`
- `melody_path/chord_path` 继续负责存储 MIDI 产物路径
- `melody_key/chord_key` 负责建立歌曲与分析缓存记录的稳定关联

## 4.2 分析缓存表复用与扩展

### 方案结论
- **不新增独立新表**
- 继续复用当前 `song_analysis` 表，作为审批意见中的分析缓存表使用

### 当前已有字段
- `id`
- `song_id`
- `analysis_type`
- `result_json`
- `midi_path`
- `created_at`

### 本次建议新增字段
为满足“key 绑定”和“简谱缓存”要求，建议在 `song_analysis` 表新增：
- `analysis_key` VARCHAR(32) NOT NULL COMMENT '分析记录唯一业务key'
- `updated_at` TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间'

说明：
- `result_json` 已可承载结构化简谱内容，因此**不必额外拆出 melody_notation 字段也可满足需求**
- 若考虑 JSON 检索性能或兼容老 MySQL，也可选“简谱文件路径”方案，但从当前需求看，**优先建议把完整结构化简谱直接存到 `result_json`**，这样最利于前端直出可视化
- `analysis_key` 用于被 `songs.melody_key / songs.chord_key` 引用

### 约束建议
- 唯一约束：`UNIQUE KEY uk_analysis_key (analysis_key)`
- 查询索引：`INDEX idx_song_type (song_id, analysis_type)`

如业务要求“一首歌每种类型只有一个当前缓存”，可进一步补充：
- `UNIQUE KEY uk_song_analysis_song_type (song_id, analysis_type)`

若暂不加唯一索引，则 mapper 层必须实现覆盖写入策略，避免旧缓存残留。

## 5. 缓存数据结构建议

结构化简谱建议直接存入 `song_analysis.result_json`，示例：

```json
{
  "key": {
    "tonic": "C",
    "mode": "major",
    "display": "C major",
    "confidence": 0.82
  },
  "time_signature": {
    "numerator": 4,
    "denominator": 4,
    "source": "midi_default"
  },
  "tempo": {
    "bpm": 112.3
  },
  "duration": {
    "seconds": 185.4
  },
  "notation": {
    "mode": "numbered",
    "notes": [
      {
        "start": 0.52,
        "end": 0.78,
        "midi": 60,
        "pitch_name": "C4",
        "degree": "1",
        "octave_offset": 0,
        "duration_beats": 0.5
      }
    ],
    "measures": [
      {
        "index": 0,
        "start": 0.0,
        "end": 2.14,
        "degrees": ["1", "2", "3", "-"]
      }
    ]
  }
}
```

说明：
- 调性信息保存在 `result_json.key` 中；`songs.melody_key` 保存的是分析缓存记录的业务 key，二者不是同一概念，不能混用
- `notation` 作为简谱缓存主体，前端无需再次推导
- 后续若扩展和弦分析，也可沿用同一表，仅以 `analysis_type='chord'` 区分

## 6. API 契约（按审批意见修订）

### 6.1 查询旋律分析缓存

- **路径**：`/api/songs/<song_id>/melody-analysis`
- **方法**：`GET`
- **说明**：仅用于读取缓存，不触发重新解析

### 6.2 成功响应

```json
{
  "code": 200,
  "description": "success",
  "result": {
    "song": {
      "id": 123,
      "melody_key": "melody_123_abcd1234"
    },
    "analysis": {
      "type": "melody",
      "midi_path": "transcribe/123_melody_task_xxx.mid",
      "key": {
        "tonic": "C",
        "mode": "major",
        "display": "C major",
        "confidence": 0.82
      },
      "time_signature": {
        "numerator": 4,
        "denominator": 4,
        "source": "midi_default"
      },
      "tempo": {
        "bpm": 112.3
      },
      "duration": {
        "seconds": 185.4
      },
      "notation": {
        "notes": [
          {
            "start": 0.52,
            "end": 0.78,
            "midi": 60,
            "pitch_name": "C4",
            "degree": "1",
            "octave_offset": 0,
            "duration_beats": 0.5
          }
        ],
        "measures": [
          {
            "index": 0,
            "start": 0.0,
            "end": 2.14,
            "degrees": ["1", "2", "3", "-"]
          }
        ]
      }
    }
  }
}
```

结构确认结论：
- **保持当前嵌套结构，不改回前端早期文档里的平铺格式**
- `result.song` 负责承载歌曲级元信息（当前最少需要 `id`、`melody_key`）
- `result.analysis` 负责承载分析结果本体（调性、拍号、速度、时长、简谱内容）
- 这种分组方式比 `song_id / key / notes / measures` 平铺在 `result` 下更稳定，后续若增加 `analysis_key`、版本信息、缓存时间、和弦分析复用字段时，不会污染同一层级
- 前端如只关心渲染数据，可直接读取 `result.analysis.notation.notes`、`result.analysis.notation.measures`、`result.analysis.key` 等字段，不会增加明显解析成本

符合审批要求：
- 顶层统一为 `code / description / result`
- 业务字段全部收纳到 `result` 内
- `result` 内继续按对象分组，不平铺顶层字段
- 因此前后端对齐时，应以本嵌套结构作为最终契约

### 6.3 错误响应建议

#### 歌曲不存在 `404`
```json
{
  "code": 404,
  "description": "Song not found",
  "result": null
}
```

#### 旋律分析缓存不存在 `404`
```json
{
  "code": 404,
  "description": "Melody analysis not found",
  "result": null
}
```

#### 缓存损坏或读取失败 `500`
```json
{
  "code": 500,
  "description": "Failed to load melody analysis",
  "result": null
}
```

说明：
- 本次统一包装要求建议成功、失败都遵循同一响应骨架
- 但审批意见明确强调的是成功格式；若项目已有全局错误格式约束不同，则此项需在实现前再确认

## 7. 数据流调整

### 7.1 写入链路
1. 前端/任务触发旋律转录
2. 后端生成 MIDI 文件
3. 后端解析 MIDI，构建结构化旋律分析结果
4. 写入 `song_analysis`：
   - `analysis_type = 'melody'`
   - `analysis_key = <生成的唯一key>`
   - `result_json = <结构化简谱 JSON>`
   - `midi_path = <旋律MIDI路径>`
5. 回写 `songs.melody_key = analysis_key`
6. 后续查询接口直接读缓存

### 7.2 查询链路
1. 前端请求 `GET /api/songs/<song_id>/melody-analysis`
2. 后端查询 `songs`
3. 使用 `songs.melody_key` 或 `song_id + analysis_type='melody'` 查询分析缓存
4. 返回统一包装后的 `result`
