# basic-pitch后处理优化需求分析

## 1.1 需求目标

在现有“demucs 先做人声分离，再用 basic-pitch 基于人声文件提取 MIDI”的旋律提取主路径上，进一步提升 MIDI 听感与旋律保真度。

本次后端目标聚焦三类优化：

1. 修复 `BasicPitchMelodyTranscriber._filter_melody_only` 中的音符裁剪缺陷，避免重叠音符被整段误丢弃。
2. 提升 MIDI 输出表现力，包括：
   - 基于输入音频检测 tempo，替代固定 120 BPM
   - 对音符 velocity 做归一化，提升动态表现
3. 调整 `basic_pitch` 默认参数，使人声软起音、短时值音符场景下的识别效果更稳定。

本次需求属于对既有旋律提取链路的后处理与调参增强，不涉及架构级路线切换。

| config.json | 修改 | 调整 `basic_pitch` 默认参数，并新增 tempo/velocity 相关配置 |
| mapper/repository | 无改动 | 本次不涉及数据库持久化结构 |

## 1.6 API 兼容性影响

当前判断：**不影响已有接口调用方**。

原因：
- 本次优化聚焦于 MIDI 生成逻辑与配置默认值。
- 若接口路径、方法、请求参数、响应格式均不变，则前端只会感知到“生成结果更好”，不会产生调用兼容性问题。

需要同步关注的点：
- 若前端依赖某些固定输出节奏或音符分布做展示，需确认优化后是否会影响展示逻辑。
- 若后续要开放参数调优入口给前端，则需另行设计接口扩展。

## 1.7 风险点 / 依赖项 / 待确认项

🚨 风险：
- `_filter_melody_only` 改为扫描线实现后，逻辑正确性提升，但需要重点验证不会引入新的音符切分异常。
- tempo 检测在强节拍弱、节奏模糊的人声素材上可能出现 half/double 误判。
- velocity 归一化会改变 MIDI 播放器中的主观听感，若归一化范围设得过激，可能导致弱音被抬得过响。
- `minimum_note_length_ms` 与 `onset_threshold` 下调后，可能出现“切得更碎”或引入更多杂音符。

🔗 依赖：
- 依赖现有 `basic_pitch` 路径已稳定集成，并继续作为默认旋律提取主路径。
- 依赖现有测试音频/样例歌曲可重复回归，用于对比修复前后结果。
- 依赖项目当前环境已具备 `librosa` 与 basic-pitch 所需运行条件。

❓ 待确认：
- 当前 `config.json` 中 `basic_pitch` 的真实配置路径与键名，是否与方案文档中的示例完全一致。
- 当前 transcribe 接口响应里，是否需要增加调试信息（例如 tempo、total_notes、过滤前后数量）。
- 项目目前是否已有正式测试目录；若没有，本次是否允许先用临时脚本验证 P0 overlap 用例。

## 3.1 API 契约设计

### API：旋律提取任务接口
- **接口类型：** 复用无改动
- **用途：** 继续用于上传或指定音频后触发旋律提取，输出 MIDI 结果
- **路径：** 以现有 `transcribe_controller` 实际路由为准
- **方法：** 以现有实现为准
- **鉴权：** 当前项目默认不需要

**请求参数：**
- 本次需求不新增参数。
- 若未来需支持前端指定 `detect_tempo`、`velocity_min/max` 等调优项，应单独评估是否开放给前端，避免把内部实验参数直接暴露到正式接口。

**成功响应（200）：**
- 保持现有结构不变。
- 重点变化在生成结果质量，而非接口契约。

**错误响应：**
- 保持现有结构不变。
- 建议补充日志：当 BPM 检测失败时，不中断主流程，记录 warning 后回退默认 tempo。

## 4.1 完整请求-响应链路

```
用户上传/选择音频
   │
   ▼
前端发起旋律提取请求
   │
   ▼
transcribe_controller 接收请求并校验参数
   │
   ▼
转录业务层/编排逻辑调用旋律提取器
   │
   ▼
DemucsMelodyTranscriber 先做人声分离
   │
   ▼
BasicPitchMelodyTranscriber 基于人声文件执行音符识别
   │
   ├─ 识别后执行 melody_only 重叠音符过滤
   ├─ 执行音符去重与排序
   ├─ 执行 velocity 归一化
   ├─ 检测 tempo（失败则回退固定值）
   └─ 输出 MIDI 文件
   │
   ▼
控制器返回任务结果/文件路径/状态信息
   │
   ▼
前端播放或下载 MIDI，感知优化后的旋律与节奏表现
```

## 实施建议（后端视角）

### 优先级建议
1. **P0：** 先修复 `_filter_melody_only` 漏音符 bug
2. **P1：** 再补 tempo 检测与 velocity 归一化
3. **P2：** 最后调整默认配置参数，并做端到端听感回归

### 建议改动范围
- 代码主改动：`backend/transcriber/basic_pitch/melody.py`
- 配置改动：`backend/config.json`
- 测试验证：优先增加可复用测试；若项目暂未建立标准测试结构，可先用一次性验证脚本辅助确认，再补正式测试

### 不建议本次一并处理的事项
- basic-pitch 直接处理原曲的 A/B 架构实验
- 八度错修正
- pitch_bend 写入 MIDI
- 前端参数调节 UI

这些事项都应视为独立需求，避免本次需求范围膨胀。

## 结论

我认为这次需求在后端侧属于**低风险、可控范围内的结果质量增强**，且不涉及数据库和现有 API 契约变更，适合作为独立小迭代推进。

建议先按照“P0 bug 修复 → P1 输出表现增强 → P2 参数微调”的顺序实施，并以同一首已知测试歌曲做修复前后对比，确保听感提升真实可验证。
